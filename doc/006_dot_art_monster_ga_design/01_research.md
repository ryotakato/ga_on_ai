# 調査: ドット絵モンスターGAの設計

## 背景・目的

既存の ga_on_ai システムに、新しい遺伝子タイプとして**ドット絵モンスター**を追加したい。

- `all_genes.json` の `content` フィールドにドット絵を文字列で表現する
- 評価基準：「ゲームのモンスターっぽいかどうか」
- 既存の gene-generator-agent / gene-evaluator-agent の仕組みをできる限り流用する

---

## 論点1: ドット絵の文字列表現方法

### 案A: 文字グリッド方式

各ピクセルを1文字で表現し、行を `\n` で区切る。

```
........
..GGGG..
.GGYYGG.
.GY..YG.
.GGGGGG.
..RRRR..
........
```

- `.` = 透明/背景
- `R`, `G`, `Y`, `B` などのアルファベット = 色（限定パレット）
- 16×16 または 8×8 が現実的なサイズ

**長所:**
- LLMが読み書きしやすい
- レンダラーの実装が単純（1文字 → 1ピクセル）
- 既存の `content` フィールドにそのまま収まる

**短所:**
- 機械的交叉で空間的一貫性が崩れやすい（後述）

### 案B: 描画命令方式

モンスターの形状を描画命令の列として表現する。

```
bg=black
circle(4,3,3,green)
eyes(3,3,red,2)
mouth(4,5,white)
spikes(top,3,yellow)
```

**長所:**
- 命令単位での交叉が可能 → 「色だけ親Bから」「形は親Aから」など自然な継承
- スライムなど非構造的モンスターも `blob(5,5,4,purple)` 一命令で表現できる
- LLMが初期世代の生成に適している

**短所:**
- レンダラーの実装が複雑（命令セットの定義が必要）
- 命令セットの表現力の限界がモンスターの多様性を制限する

---

## 論点2: 掛け合わせ（交叉）の難しさ

ドット絵の交叉は文字列（小説・なぞかけ）と根本的に異なる。
**ピクセルは隣のピクセルと空間的に連続した意味を持つ**ため、単純な文字列混合では破綻しやすい。

### 機械的交叉の問題

| 手法 | 問題 |
|------|------|
| 行ベース分割（上半分＋下半分） | 継ぎ目が水平線として明確に見える |
| 列ベース分割 | 継ぎ目が垂直線として見える |
| チェッカーボード | ノイズ状になる |
| ランダムマスク（各ピクセル独立） | 塩胡椒ノイズ状になる |

→ **「上半分は炎のモンスター、下半分は水のモンスター」のような継ぎ目が目立つ**

ただし、機械的交叉で生まれた不自然な個体は評価で淘汰され、世代を重ねるうちに収束する可能性はある（実験的に確認が必要）。

### 構造化交叉の問題

「頭・胴体・手足」などのパーツ単位で交叉する方式。

→ **スライム・アメーバ・目玉など、身体図式を持たないモンスターが生まれない**という致命的制約がある。GAで多様性を担保したい場合には不向き。

### LLM任せの交叉の問題

「この2つの親ドット絵を掛け合わせて新しいものを作れ」とLLMに指示する方式。

→ **LLMが文脈からモンスター生成という目標を認識してしまう**。これは `gene-generator-agent` の設計原則（generatorは評価目標を知らない）に反する。GAとして純粋に機能しない。

### 描画命令方式での交叉

命令列として表現されていれば、命令単位での機械的な入れ替えが可能。

```python
# 例: 親Aの命令3つ + 親Bの命令2つ
parent_a = ["bg=black", "circle(4,3,3,green)", "eyes(3,3,red,2)"]
parent_b = ["bg=purple", "blob(5,5,4,blue)", "mouth(4,5,white)", "spikes(top,3,yellow)"]
child = parent_a[:2] + parent_b[2:]
# → ["bg=black", "circle(4,3,3,green)", "mouth(4,5,white)", "spikes(top,3,yellow)"]
```

空間的な継ぎ目は生じにくいが、命令間の整合性が崩れる可能性はある。

## 論点3: 評価方法

### 案A: テキスト表現のまま評価

`content` の文字グリッドまたは描画命令をそのままLLMが読んで評価する。

**問題:** 8×8〜16×16のキャラクターグリッドからモンスターの視覚的印象を読み取るのは困難。評価精度に限界がある。

### 案B: 画像レンダリングして評価（推奨）

`render_sprite_sheet.py` を用意し、評価前に all_genes.json を1枚のスプライトシートPNGに変換。evaluatorがその画像を Read して視覚的に評価する。

```
render_sprite_sheet.py 実行
  → 100体を10×10グリッドで1枚のPNGに並べる（gene_numラベル付き）
  → evaluator が1枚の画像を Read して一括ランキング
```

**トークンコスト:**
- 100体を個別に渡す場合：画像100枚分（高コスト）
- スプライトシート1枚にまとめる場合：画像1枚分（現行のテキスト評価と同程度のコスト増）

**長所:**
- LLMが視覚的にモンスターを判断できる → 評価精度が大幅に向上
- evaluator-agent の tools（Read, Write, Bash）は**変更不要**
- ワークフローの変更は「評価前にスクリプトを呼ぶ1ステップ追加」だけ

---

## 追加コンポーネントの整理（スプライトシート案の場合）

| コンポーネント | 内容 | 実装難易度 |
|--------------|------|-----------|
| `render_sprite_sheet.py` | all_genes.json → スプライトシートPNG（Pillow使用） | 低 |
| `gene-evaluator-agent.md` | Step2の前にスクリプト呼び出しを追加 | 低 |
| `dotmonster/criteria.md` | 新遺伝子タイプの設定・評価基準定義 | 低 |
| 交叉スクリプト（オプション） | 機械的交叉をPythonで実装する場合 | 中 |

---

## 暫定推奨アプローチ

**フェーズ1: 文字グリッド表現 ＋ スプライトシート評価**

1. `content` = 文字グリッド（16×16、限定パレット）
2. 交叉 = 機械的（行ベースなど）またはLLMが「グリッドを見て新しいモンスターを描く」方式を試す
3. 評価 = スプライトシートPNGを生成してevaluatorが視覚的に判定
4. 数世代試して交叉の品質を観察する

**フェーズ2: 問題があれば描画命令方式に移行**

- 機械的交叉の継ぎ目問題が顕著であれば描画命令方式に切り替え
- レンダラーを実装し直す

## 未解決の課題

1. **文字グリッドの仕様**: サイズ（8×8 / 16×16）、パレット（何色まで）
2. **描画命令の命令セット定義**: どの命令をサポートするか（circle, rect, line, blob, eyes, mouth, spikes...）
3. **機械的交叉の具体アルゴリズム**: どの手法が最も自然な結果を生むか（実験が必要）
4. **初期世代の生成**: LLMに「ゲームモンスター風のドット絵を文字グリッドで描け」と指示する際のプロンプト設計
5. **render_sprite_sheet.py の仕様**: パレット定義、ピクセルサイズ、ラベルの配置
