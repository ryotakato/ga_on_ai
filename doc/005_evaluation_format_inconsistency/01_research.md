# 調査: novel140 evaluation.md フォーマット不一致の原因

## 観察された事実

novel140 の evaluation.md に2種類のフォーマットが混在している。

### 旧フォーマット（世代001・002）

```
| 順位 | id | テキスト（冒頭30字） | 総評 |
|------|-----|---------------------|------|
| 1 | 25 | 百歳の女性に長寿の秘訣を聞いた。 | 完結性◎ 感情◎ 独自性◎ |
```

特徴:
- 列が4列（順位 / id / テキスト冒頭30字 / 総評）
- `id` 列が**ゼロパディングなし整数**（例: `25`、`50`）
- `gene_num` という列名が存在しない

### 新フォーマット（世代003・004・005）

```
---
generation: 003
evaluated_at: 2026-02-26
---

| rank | gene_num |
|------|----------|
| 1    | 042      |
| 2    | 030      |
```

特徴:
- 列が2列（rank / gene_num）
- gene_num が**ゼロパディングあり**（例: `042`）
- YAMLフロントマターあり

---

## 影響: select_parents.py との互換性

`select_parents.py` のパーサーは `| rank | gene_num |` 形式を前提としている。

旧フォーマットに対して実行した場合:
- 列名は `\|\s*(\d+)\s*\|\s*(\w+)\s*\|` で正規表現的には一致するが、取得される `gene_num` が `"25"` になる
- `all_genes.json` 側のキーは `"025"`（ゼロパディングあり）のため**マッチしない**
- `extract_genes.py` も `"25"` と `"025"` を別物として扱うため、実質的に選択不能

## 根本原因の特定

### 確定事実

1. `gene-evaluator-agent.md` は **最初から正しいフォーマットを定義していた**
2. novel140/001・002 の評価は **`gene-evaluator-agent` を経由して実行された**（Claude Code の実行履歴で確認）
3. novel140/001 の all_genes.json 内の gene_num は `"001"`, `"025"` など**ゼロパディングあり3桁**
4. novel140/001 の evaluation.md 内の id は `25`, `64` など**ゼロパディングなし整数** → all_genes.json と不一致
5. ジェネレーターでは同様のフォーマット上書き問題が確認されており、second commit で「固定」注釈が追加された

### ~~当初の仮説（誤り）~~

~~novel140/001・002 はインタラクティブな Claude 会話で手動評価された~~

→ 実行履歴により否定。evaluator-agent 経由で評価されていた。

### 正しい結論

**`gene-evaluator-agent` がフォーマット定義を持っていたにもかかわらず、オーケストレーター（SKILL.md を実行する LLM）が渡した指示によってフォーマットが上書きされた。**

これは generator で起きたことと同じメカニズムである:

| エージェント | フォーマット上書き問題 | 「固定」保護注釈 |
|-------------|----------------------|----------------|
| generator | 発生 → second commit で修正 | **あり** |
| evaluator | 発生（001・002）→ 003以降は正常 | **なし（未対処）** |

### なぜ003以降は正しいフォーマットになったか

second commit（d02c1ce）で generator に「固定」注釈が追加されたのと同タイミングで、SKILL.md に以下が追加された:

```
**注意: プロンプトにファイルフォーマットや出力形式を追加してはならない。フォーマットはサブエージェント自身が定義を持っている。**
```

ただしこの注釈は **generator 起動セクションのみ** に追加され、evaluator 起動セクションには追加されなかった。にもかかわらず 003 以降は正しいフォーマットになっているのは、この注釈が存在することでオーケストレーター LLM の「フォーマットを上書きしない」という意識が全体的に機能した可能性がある。

### 現在のリスク

evaluator-agent 本体に「フォーマット固定」保護注釈がない状態が続いている。
generator と同様に明示的な保護を追加しないと、再発リスクが残る。
