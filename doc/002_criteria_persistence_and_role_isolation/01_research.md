# 調査レポート: 基準のファイル永続化とジェネレーター/エバリュエーター間の情報分離

調査日: 2026-02-22

---

## 1. 問題の整理

### 1-1. 現在の問題

現在、遺伝子の「掛け合わせ方」と「評価基準」はプロンプト入力に依存している。

```
ユーザーがスキル起動時に指示 → セッション内のみ有効
     ↓
セッションを閉じると消える → 毎回再入力が必要
```

具体的には SKILL.md に以下のような記述がある：

**gene-generator/SKILL.md（抜粋）:**
```
子のファイルを作るときの、遺伝子の中身の掛け合わせ方は、このスキルを指示されたときのやり方に従う。
指示されていなかったら、確認する。
```

**gene-evaluator/SKILL.md（抜粋）:**
```
全ての遺伝子の中身を一度に読んだ上で、このスキルが指示されたときの判断基準に則って評価し、
頭の中でランキングを決定する。
```

### 1-2. ユーザーの制約条件

| 制約 | 内容 |
|------|------|
| ファイル数 | 遺伝子タイプごとに2ファイルは多い。1ファイルにしたい |
| 将来拡張 | 現在はなぞかけのみ。今後複数の遺伝子タイプが増える予定 |
| 運用 | 現在は別ターミナル2つで動かしている。できれば1セッションに統合したい |
| 情報分離 | GeneratorはEvaluatorの基準を知らない方が望ましい（逆も同様） |

---

## 2. なぜ情報分離が重要か

遺伝的アルゴリズムにおける分離の意味を整理する。

### 2-1. 分離が必要な理由

```
Generatorが評価基準を知っている場合：
  → 「笑える」ことが評価基準だとわかれば
  → 親遺伝子を無視して、笑えるなぞかけを直接生成してしまう可能性
  → これは「進化」ではなく「最適化」

Evaluatorが掛け合わせ方を知っている場合：
  → 評価に影響はないが、不要な情報がコンテキストを汚染する
```

### 2-2. 分離の効果の現実的な評価

ただし、現行のGAシステムの構造を踏まえると、分離の重要度には留意点がある：

```
現行構造：
  ランダムに親2個を選ぶ（gen-generatorのStep 4）
     ↓
  掛け合わせ方の指示に従って子を生成
```

親の選択がランダムである以上、仮にGeneratorが評価基準を知っていても、
選ばれた親に縛られる制約の方が大きい。純粋な「評価チート」になりにくい。

しかし：
- 将来的に「選択的な掛け合わせ」（強い親を意識した生成）を導入する場合、この分離は重要になる
- 「情報の清潔さ」としての設計方針としても有意義

**結論**: 現時点では分離の実質効果は限定的だが、設計方針として分離を維持する価値はある。

## 3. 解決策の候補

### 候補1: 1ファイル + セクション分割 + Bashによる抽出

**ファイル構造：**
```
criteria/
  nazokake.md
  （将来の遺伝子タイプ）.md
```

**ファイルフォーマット：**
```markdown
---
gene_type: nazokake
---

## crossbreeding

（掛け合わせ方の説明）

## evaluation

（評価基準の説明）
```

**分離の仕組み：**

各SKILL.md内で、Bashを使って必要なセクションだけを抽出する。

```bash
# gene-generatorが使う（crossbreedingセクションだけを取得）
awk '/^## crossbreeding$/,/^## /' criteria/nazokake.md | grep -v '^## evaluation'
```

```bash
# gene-evaluatorが使う（evaluationセクションだけを取得）
awk '/^## evaluation$/,/^## /' criteria/nazokake.md
```

**評価：**

| 観点 | 評価 |
|------|------|
| ファイル数 | 1ファイル/タイプ ✓ |
| 情報分離 | Bashが構造的に分離 → 強い分離 ✓ |
| 実装コスト | SKILL.mdにBashコマンドを追加するだけ ✓ |
| 将来拡張 | 新タイプ = 新ファイル1つだけ ✓ |
| デメリット | awkコマンドがやや複雑。Claudeが正確に実行できるか |

---

### 候補2: 1ファイル + セクション分割 + 指示による軟分離

同じファイル構造だが、Bashで抽出せず、SKILL.mdに「crossbreedingセクションのみ参照すること」と指示する。

**評価：**

| 観点 | 評価 |
|------|------|
| ファイル数 | 1ファイル/タイプ ✓ |
| 情報分離 | 指示ベース → 弱い分離（Claudeの指示遵守に依存）△ |
| 実装コスト | 最も低い ✓ |
| 将来拡張 | 同上 ✓ |
| デメリット | Claudeがファイル全体を「見た上で」特定セクションを無視する、という構造。漏洩リスクは残る |

---

### 候補3: 1ファイル + サブエージェントによる分離

**仕組み：**

```
メインセッション（orchestrator）
  ├── criteria/nazokake.md を読む（全内容を把握）
  ├── crossbreedingセクションだけをプロンプトに含めてGeneratorサブエージェントを起動
  └── evaluationセクションだけをプロンプトに含めてEvaluatorサブエージェントを起動
```

各サブエージェントのコンテキストには、自分の役割に関係する基準しか含まれない。

**評価：**

| 観点 | 評価 |
|------|------|
| ファイル数 | 1ファイル/タイプ ✓ |
| 情報分離 | コンテキストレベルで完全分離 ✓✓ |
| 1セッション運用 | Taskツール追加で実現 ✓ |
| 実装コスト | メインセッション（orchestratorの役割）の設計が必要 |
| デメリット | Taskツールをallowed-toolsに追加する必要がある。orchestratorの設計が増える |

---

### 候補4: YAML構造化 + Bashによるフィールド抽出

**ファイルフォーマット：**
```yaml
---
gene_type: nazokake
crossbreeding: |
  （掛け合わせ方）
evaluation: |
  （評価基準）
---
```

**分離：**
```bash
# crossbreedingだけ取得
python3 -c "import sys,re; m=re.search(r'crossbreeding: \|(.*?)^[a-z]', open('criteria/nazokake.md').read(), re.DOTALL|re.MULTILINE); print(m.group(1).strip())"
```

**評価：**

| 観点 | 評価 |
|------|------|
| ファイル数 | 1ファイル/タイプ ✓ |
| 情報分離 | 構造的に分離可能 ✓ |
| 実装コスト | YAMLパース用コマンドが複雑 ✗ |
| デメリット | bashでYAML multiline blockを安全に扱うのは難しい |

## 5. サブエージェント（Taskツール）の動作詳細

### 5-1. サブエージェントのコンテキスト分離

Claude CodeのTaskツールは、サブエージェントを独立したコンテキストで起動する。

```
メインセッション
  会話履歴: [ユーザーとのやり取り全体]
  ↓
  Task("...", prompt="crossbreeding: ○○\n...")を起動
  ↓
サブエージェント（独立したコンテキスト）
  会話履歴: [Taskのpromptのみ]
  → evaluationセクションは存在しない
  ← 結果（生成された遺伝子100個のパス一覧など）を返す
```

**重要な特徴：**
- サブエージェントはpromptで渡された情報しか持たない
- メインセッションの会話履歴はサブエージェントに渡らない
- サブエージェントのコンテキストはメインに戻らない（結果サマリのみ）

### 5-2. サブエージェント利用時の制約

```
現状: gene-generator/SKILL.md の allowed-tools: Read, Write, Bash
      gene-evaluator/SKILL.md の allowed-tools: Read, Write, Bash
```

Taskツールを利用するには：
- オーケストレーター側（メインセッションまたはnew skill）に `Task` をallowed-toolsに追加
- もしくはSKILL.md内で `Task` を使う場合は、そのSKILL.mdのallowed-toolsに `Task` を追加

### 5-3. セッション統合の設計案

現在2ターミナル → 1セッションに統合する場合の構成：

```
新スキル: gene-evolve（オーケストレーター）
  allowed-tools: Read, Task
  ↓
  1. criteria/{gene_type}.md を読む
  2. crossbreedingセクション → Generatorサブエージェントに渡す
  3. Evaluatorサブエージェントを起動
  4. evaluation.md 完成後、次世代サイクルへ
```

または、既存スキルを維持しつつ：

```
ユーザー: 「進化させて（評価基準: ○○、掛け合わせ: ○○）」
     ↓（基準はファイルから読む）
gene-generator スキル
  SKILL.md に「criteriaファイルからcrossbreedingセクションをBashで抽出して使う」と記載
```

## 6. 推奨方針

### 6-1. 基本方針

**候補1（Bash抽出）をベースとし、将来的に候補3（サブエージェント）へ移行**

**フェーズ1（今すぐできる）：**
- `criteria/` ディレクトリを作成
- 遺伝子タイプごとに1ファイル（`nazokake.md` など）
- ファイル内をセクション分割（`## crossbreeding` / `## evaluation`）
- 各SKILL.mdを修正し、Bashでセクション抽出して使用
- 分離の強度: 中（構造的だが同プロセス内）

**フェーズ2（1セッション化したいとき）：**
- オーケストレーター用スキルまたは既存スキルにTaskツールを追加
- criteriaファイルを読んでから、セクションごとにサブエージェントに渡す
- 分離の強度: 最強（コンテキストレベル）

### 6-2. criteriaファイルの推奨フォーマット

```markdown
---
gene_type: nazokake
---

## crossbreeding

（掛け合わせ方の説明をここに書く）

## evaluation

（評価基準の説明をここに書く）
```

### 6-3. ディレクトリ構造

```
generate/
  criteria/
    nazokake.md         ← 現在の遺伝子タイプ
    （将来のタイプ）.md  ← 増えても1ファイル/タイプ
  gene/
    ...（既存）
  .claude/
    skills/
      gene-generator/
      gene-evaluator/
```
