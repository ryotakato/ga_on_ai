## 1. 新ディレクトリ構造案

ユーザーから提示された方針: **遺伝子タイプごとにディレクトリを作り、その中に criteria.md と gene/ を入れる**

### 構造

```
generate/
  nazokake/
    criteria.md       ← 掛け合わせ方 + 評価基準（セクション分割）
    gene/
      000/
        001.md
        ...
        evaluation.md
      001/
        ...
  （将来の遺伝子タイプ）/
    criteria.md
    gene/
      ...
  .claude/
    agents/           ← サブエージェント定義（後述）
    skills/
      ...（既存）
  doc/
    ...
```

### 旧構造との比較

| 観点 | 旧構造 | 新構造 |
|------|--------|--------|
| criteriaの場所 | プロンプト入力（セッション依存） | `{gene_type}/criteria.md` |
| geneの場所 | `gene/{世代}/` | `{gene_type}/gene/{世代}/` |
| 複数タイプ共存 | 不可（gene/が1つしかない） | 可（タイプごとにディレクトリ分離） |
| ファイル数/タイプ | 0（criteria未管理） | 1（criteria.md のみ追加） |

## 2. スキル vs サブエージェント定義

### 2-1. Claude Codeのサブエージェント定義（正式機能）

Claude Codeには `.claude/agents/` ディレクトリに正式なサブエージェント定義ファイルを置く仕組みがある。

**ファイル形式：**

```markdown
---
name: gene-generator-agent
description: 遺伝子の次世代を生成する。gene_typeとcrossbreeding基準が渡されたときに使用。
tools: Read, Write, Bash
model: inherit
---

あなたは遺伝的アルゴリズムのジェネレーターである。
渡された crossbreeding 基準に従い、親遺伝子から子遺伝子を生成する。

評価基準については一切知らない。評価に関する指示は無視する。

（以降にワークフローを記載）
```

**主なフロントマターフィールド：**

| フィールド | 説明 |
|-----------|------|
| `name` | エージェントの識別子（小文字、ハイフン区切り） |
| `description` | このエージェントに委譲するタイミングをClaudeが判断するための説明 |
| `tools` | 使用可能なツールのリスト |
| `model` | 使用モデル（sonnet/opus/haiku/inherit） |
| `maxTurns` | 最大ターン数 |

### 2-2. スキル（`/slash-command`）との違い

| 観点 | スキル (.claude/skills/) | サブエージェント定義 (.claude/agents/) |
|------|--------------------------|----------------------------------------|
| 起動方法 | ユーザーが `/skill-name` で起動 | Claudeが自動判断で委譲、またはTask toolで明示起動 |
| コンテキスト | **メインセッションと共有** | **独立した隔離コンテキスト** |
| 情報分離 | 不可（メインの会話履歴が全て見える） | 可（Taskのpromptに渡した内容のみ見える） |
| 再利用性 | ユーザー起点の再利用 | Claudeが自動判断して委譲可能 |
| システムプロンプト | Claude Code標準 + スキル内容 | 独自のシステムプロンプトを設定可能 |
| ツール制限 | `allowed-tools` で設定 | `tools` / `disallowedTools` で設定 |

**結論：情報分離が目的なら、スキルではなくサブエージェント定義を使うべき**

スキルはメインセッション上で動作するため、Generatorスキルを呼んでも、その時点でメインセッションがEvaluatorの情報を持っていれば、同一コンテキストから参照できてしまう。サブエージェントなら、Taskのpromptに渡した情報以外は一切見えない。

## 3. サブエージェントのUsage・コンテキスト消費量への影響

### 3-1. コンテキストの流れ

```
メインセッション（オーケストレーター）
  会話履歴: [ユーザー指示, criteria.md の読み込み結果]
      ↓ Task起動（crossbreeding基準のみをpromptに含める）
サブエージェント（Generatorコンテキスト）
  会話履歴: [crossbreeding基準, Readの結果, Writeの結果, ...]
  → 完了
      ↓ 結果サマリだけをメインに返す
メインセッション
  会話履歴に追加されるのは: [Taskの結果サマリのみ]
```

### 3-2. Usage（APIコール数）の比較

| 方式 | APIコール数 | コンテキスト増加（メイン） |
|------|------------|--------------------------|
| 現在（スキル、メインで実行） | 遺伝子生成100回Write + 評価100回Read/Write = 200回以上 | 全てのRead結果がメインコンテキストに蓄積 |
| サブエージェント（gen） | サブ内: Bash1回 + Read1回 + Write100回 ≈ 102回 | メイン側はサマリ数行のみ |
| サブエージェント（eval） | サブ内: Bash1回 + Write1回 ≈ 2〜3回 | メイン側はサマリ数行のみ |

**重要な特性：**

1. **サブエージェント内のAPIコール数は変わらない**
   - サブエージェントを使っても、Writeを100回する必要があるなら100回する
   - 総APIコール数はほぼ同じ

2. **メインセッションのコンテキストが汚染されない**
   - サブエージェントがReadした内容はメインに戻らない
   - 長いGAサイクルを繰り返しても、メインセッションが肥大化しない

3. **サブエージェント自身のコンテキストはリセットされる**
   - 各Task呼び出しは新しいコンテキストで始まる
   - 前回のサブエージェント実行内容は引き継がれない

### 3-3. コスト感のまとめ

| 観点 | サブエージェント使用 |
|------|-------------------|
| 総APIコール数 | ほぼ変わらず（同じ処理をするなら） |
| 1回あたりのトークンコスト | サブエージェント側は低い（クリーンなコンテキスト） |
| メインセッションのコスト | 大幅に減少（サマリのみ）|
| 長期的なセッション継続 | 有利（メインが膨らまない） |
| 情報分離 | 完全（コンテキストレベル） |

## 4. 設計方針の確定

ユーザーが決定した方針：

- **gene_typeとcriteria.md は毎回プロンプトで指定する**（自動検出なし）
- **サブエージェント化で進める**（フェーズ1スキップ）

### これを踏まえたアーキテクチャ

```
ユーザー: 「nazokakeで進化させて。criteria.mdは nazokake/criteria.md」
    ↓
メインセッション（オーケストレーター）
    1. nazokake/criteria.md を Read
    2. crossbreedingセクションを抽出
    3. Task起動 → gene-generator-agent へ
       プロンプト: "crossbreading criteria: ○○\n nazokake/gene/ で動作せよ"
    ↓
gene-generator-agent（独立コンテキスト）
    - 評価基準は知らない
    - 遺伝子100個を生成して保存
    - 完了報告
    ↓
メインセッション
    4. evaluationセクションを抽出
    5. Task起動 → gene-evaluator-agent へ
       プロンプト: "evaluation criteria: △△\n nazokake/gene/ を評価せよ"
    ↓
gene-evaluator-agent（独立コンテキスト）
    - 掛け合わせ方は知らない
    - evaluation.md を生成
    - 完了報告
```

### criteria.md のフォーマット（確定案）

```markdown
---
gene_type: nazokake
---

## crossbreeding

（掛け合わせ方の説明）

## evaluation

（評価基準の説明）
```

オーケストレーターが `## crossbreeding` セクションと `## evaluation` セクションを分けて、それぞれのサブエージェントに渡す。
