# 調査: 遺伝子タイプごとの選択手法設計

## 背景・きっかけ

novel140（140字小説GA）の第2世代を評価した際、上位に「ねこさがしています」モチーフが3本並ぶという現象が発生した。これは**早期収束**（premature convergence）の典型例であり、現行の「上位20個のみをエリートとして選択する」方式に起因する。

現在の `criteria.md` パラメータ:
```yaml
selection_count: 20  # 上位N個のみが親候補
```

この `selection_count` だけでは選択の「方式」を指定できない。遺伝子タイプの性質（多様性重視か品質重視か）によって最適な選択手法が異なるため、**`criteria.md` に選択手法を定義し、ジェネレーターがそれに従う設計**が望ましいという結論に至った。

---

## 主要な選択手法の整理

### 1. 純粋エリート選択（現行）

上位N個のみを親プールとして使用する。

```yaml
selection_method: elite
selection_count: 20
```

**長所:**
- 品質の高い遺伝子が確実に次世代に影響する
- シンプルで実装が簡単

**短所:**
- 多様性が急速に失われる（早期収束）
- 評価が高いジャンル・スタイルに偏る
- novel140 でいえば「感動系」「老人×孤独」に収斂しやすい

**適するケース:** 収束速度を優先したい場合、ジャンルの幅より品質の洗練を求める場合

---

### 2. エリート＋ランダム混合

上位N個（エリート）＋下位からランダムM個を親プールに加える。

```yaml
selection_method: elite_plus_random
selection_count: 10   # エリート枠
diversity_count: 10   # ランダム枠（残りの中からランダム抽出）
```

**長所:**
- 実装がシンプル
- 多様性と品質のバランスが明示的に調整できる
- 「面白いが評価されにくいジャンル」（ホラー・不条理・SF）の生き残りを許容

**短所:**
- ランダム枠は純粋にランダムなので、極端に低品質な遺伝子も混入しうる

**適するケース:** novel140 のような創作系。ジャンルの多様性を保ちたい場合。現時点で最も推奨しやすい手法。

---

### 3. ランク比例確率選択（ルーレット選択）

順位に応じた確率で選択する。1位は最も高い確率、最下位でもゼロではない。

```yaml
selection_method: rank_proportionate
selection_pool_size: 100   # 全体を親プール候補にする
```

**確率の計算例（線形ランク選択）:**
- N個の遺伝子に対し、1位の重みを `2`、最下位を `0` とし線形に割り当てる
- 重みを合計で割り確率化

**長所:**
- 多様性を保ちつつ品質も維持できる
- パラメータチューニングが少ない（全体プールから確率的に選べばよい）

**短所:**
- 「確率的に選ぶ」処理をAIサブエージェントに指示するのが難しい（乱数生成の正確な実装が必要）
- 低ランク遺伝子が親になる確率は低いが、ゼロではないため品質低下リスクがある

**適するケース:** バランス型。ただし実装難易度がやや高い。

---

### 4. トーナメント選択

親プール全体からランダムにK個を取り出し、その中の最高ランクを親として採用する。これをoffspring_count回繰り返す。

```yaml
selection_method: tournament
tournament_size: 5   # 毎回K個からベストを選ぶ
```

**長所:**
- tournament_size で選択圧を細かく調整できる（大きいほどエリート寄り）
- 全体プールを使うため多様性が保たれる
- 実装がシンプルで再現性が高い

**短所:**
- パラメータ（tournament_size）の意味が直感的でない

**適するケース:** 選択圧を細かく調整したい場合。nazokakeのような「構造の精度が厳しく問われる」ジャンルに向く。

---

### 5. ニッチング / 多様性ペナルティ

似た遺伝子が多いとその遺伝子の評価スコアを下げる（または選択確率を下げる）。

**長所:**
- 意味的に似た遺伝子群（例：「ねこさがしています」系）を自動的に間引ける

**短所:**
- 「似ている」の定義が難しい（テキスト類似度の計算が必要）
- LLMによる判定は重くコストが高い

**適するケース:** 理想的だが実装コストが高い。将来的な発展形として検討余地あり。

---

## criteria.md への設計案

選択手法を `criteria.md` に定義するフィールド案:

```yaml
---
gene_type: novel140
initial_count: 100
selection_count: 10       # エリート枠（全手法共通）
diversity_count: 10       # ランダム枠（elite_plus_random の場合）
selection_method: elite_plus_random  # 選択手法
offspring_count: 100
---
```

### gene_type 別の推奨設定

| gene_type | 推奨手法 | 理由 |
|---|---|---|
| novel140 | `elite_plus_random` | ジャンル・テーマの多様性が重要 |
| nazokake | `elite` または `tournament` | 掛け言葉の精度という明確な品質基準がある |

---

## ジェネレーターへの影響

`gene-generator-agent.md` の親選択ステップ（現在: 「上位selection_count個からランダムに2個選ぶ」）を以下のように拡張する必要がある:

1. `criteria.md` から `selection_method` を読む
2. 手法に応じて親プールの構築方法を変える
3. 親プールからランダムに2個選んで交叉する（この部分は変わらない）

`selection_method` が未定義の場合は `elite`（後方互換）とする。

## 追記: 第3世代観察後の問題意識（2026-02-26）

### 掛け合わせによる物語破綻

第3世代を観察した結果、「たまに面白いものはあるが、物語として破綻しているものが増えた」という現象が確認された。

**原因の考察:**
- 初期世代（001）は Claude が1本ずつ丁寧に構成した小説のため構造が安定していた
- 掛け合わせは「象徴」を継承できるが、**物語の論理的つながり**は継承できない
- 現行の掛け合わせ方（象徴を渡すだけ）では「なぜその象徴が一つの物語に共存できるか」の必然性が担保されない

**対策候補:**
- 掛け合わせ基準に「象徴の共存理由を先に考えてから物語を書く」という制約を追加する
- 例: 「取り出した象徴が自然に同居できるテーマを先に決め、そのテーマから物語を構築する」

---

### 交叉メタデータの記録（`meta` フィールド）

どの象徴が選ばれたかが記録されていないため、破綻の原因を遡れない。また将来的に家系図と組み合わせると「どの象徴がどう受け継がれたか」を可視化できる。

**設計案: `meta` フィールドをオプションで追加**

```json
{
  "gene_num": "001",
  "content": "小説本文...",
  "parents": ["050", "023"],
  "meta": {
    "symbols": ["ねこさがしています", "初めて手をつないだ・理由は同じ"]
  }
}
```

**特徴:**
- `meta` の中身は遺伝子タイプごとに自由に定義できる（novel140 では `symbols`、他タイプでは別のキーでもよい）
- `extract_genes.py` は `gene_num` と `content` しか使わないため変更不要
- ジェネレーターへの指示に「使った象徴を `meta.symbols` に記録せよ」を追加するだけで対応可能
- `criteria.md` に `meta_fields` のような定義を追加し、ジェネレーターが参照する形にすると遺伝子タイプごとの柔軟性が保てる

**`criteria.md` への追加案:**

```yaml
meta_fields:
  - symbols   # 掛け合わせに使った象徴を記録
```
