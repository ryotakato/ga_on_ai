# gene-evaluator 高Usage問題 調査レポート

調査日: 2026-02-21

---

## 1. 現状の構造

### ファイル数

| 世代 | ファイル数 |
|------|-----------|
| 000  | 20        |
| 001  | 100       |
| 002  | 100       |
| **合計** | **220** |

### 遺伝子ファイルの内容

各ファイルは非常に小さい（10行未満）。構造は以下の通り：

```yaml
---
generation: 001
gene_num: 001
gene_eval: 42
---

（遺伝子の中身：1行程度のテキスト）
```

---

## 2. 問題の原因分析

### 2-1. gene-evaluatorのワークフロー設計

`SKILL.md` のWorkflowに「**現世代を一つ一つ評価**」と記載されている。

この指示が以下のコストを発生させる：

| ステップ | APIコール数 (100個の場合) | コストの性質 |
|----------|--------------------------|-------------|
| 各ファイルのRead | 100回 | コンテキスト蓄積（後述） |
| 各ファイルへのWrite（ランク書き込み） | 100回 | APIコール回数 |
| ランキング順位を整理するための再確認（被り禁止制約） | 追加コスト発生の可能性 | コンテキスト蓄積 |
| **合計** | **200回以上** | |

これが最大の問題。1世代100ファイルの評価で、単純に200回以上のツール呼び出しが発生する。

#### ReadとWriteでコストの性質が異なる

**Readの場合：コンテキスト蓄積が主因**

Claudeはツール呼び出しのたびにAPIコールを行い、その時点の会話履歴**全体**をトークンとして送信する。
Readはファイルの内容がそのまま会話履歴に蓄積されるため、後半になるほどコストが増大する。

```
Read file001 → 会話履歴にfile001の内容が追加
Read file002 → 会話履歴にfile001+file002の内容が含まれてAPIコール
Read file003 → 会話履歴にfile001+file002+file003の内容が含まれてAPIコール
...（100回目は99ファイル分の内容が全て含まれる）
```

**Writeの場合：APIコール回数が主因**

Writeが返すのは `"File written successfully"` 程度の確認メッセージのみ。
会話履歴への蓄積はほぼ無視できる。
ただしツール呼び出しのたびにAPIコールが走るため、**100回Writeすれば100回APIコールが発生する**。
その際の会話履歴全体の送信コストも加算される。

**まとめ：**

| ツール | 蓄積コスト | APIコール回数コスト |
|--------|-----------|-------------------|
| Read   | 大（ファイル内容が全部残る） | あり |
| Write  | 小（"success"しか残らない） | あり |

### 2-2. 「被り禁止」制約のコスト

SKILL.mdに以下の記述がある：

> 甲乙つけがたいものでも、必ず同一世代内で被ることのないようにする。

100個すべてに1〜100の被りのないランクを付けるには、Claudeが：
- 全ての評価スコアを記憶/比較し続ける必要がある
- 後のファイルを書くほど、既存のランクとの整合性確認が複雑になる
- 場合によっては前のファイルを読み直す → さらなるReadコスト

### 2-3. フォルダ構造の問題（100個の個別ファイル）

100個の小さなファイルが個別ファイルとして存在することが根本的なコスト要因。

**比較：**

| 設計 | Read回数 | Write回数 |
|------|----------|-----------|
| 現在：個別ファイル × 100 | 100回 | 100回 |
| 改善案：1世代1ファイルにまとめる | 1回 | 1回 |

各ファイルの中身はわずか数行にもかかわらず、個別ファイルのため毎回ツール呼び出しが必要。

### 2-4. スキルのallowed-tools設定

```yaml
allowed-tools: Read, Write, Bash
```

`Task` ツールが許可されていないため、サブエージェントに処理を分散できない。
全処理がメインのClaude Codeセッション（メイン会話コンテキスト）で逐次実行される。

#### サブエージェント（Taskツール）のメリットと限界

Taskツールを許可すると、Claudeはサブエージェントを別の会話コンテキストで起動できる。

```
メインコンテキスト（小）          サブエージェントコンテキスト
─────────────────                ──────────────────────────
[会話履歴: 小]                    [Read file001〜100が蓄積]
  ↓ Task起動                      [評価・ランキング処理]
  ↓ ← 結果サマリだけ受け取る ←    [結果サマリを返す]
[会話履歴: 小+サマリのみ]
```

**メリット：メインコンテキストが汚染されない**
スキル実行後も会話が軽い状態を保てる。

**限界：サブエージェント内では同じ問題が起きる**
サブエージェント自身が100ファイルをRead/Writeする構造は変わらないため、
サブエージェントのコンテキスト内での蓄積コストは解消されない。
本質的な解決策にはならない。

### 2-5. gene-generatorのフォルダ設計も同様の問題を内包

gene-generatorも100個の個別ファイルをWriteで作成するため、
次世代生成時にも100回のWriteコールが発生する。

## 3. 問題の優先度まとめ

| 問題 | 重大度 | 説明 |
|------|--------|------|
| 個別ファイル × 100 のRead/Write | **高** | 200回以上のAPIコールの直接原因 |
| 「一つ一つ評価」ワークフロー指示 | **高** | ループ処理を逐次化させる設計 |
| 被り禁止ランキングの逐次処理 | **中** | 後半ほどコストが増大する構造 |
| Taskツール不使用 | **中** | 並列化・分散ができない |
| ファイルが小さすぎる（数行）に個別管理 | **低〜中** | Read/Writeのオーバーヘッドが本文より大きい |

---

## 4. 改善の方向性（候補）

### 案A：世代ファイルの統合

各世代の遺伝子を1ファイルにまとめる。

```
gene/001.md  ← 世代001の全遺伝子を1ファイルに
```

- Read/Write: 1回で完結
- 欠点: ファイルが大きくなる（コンテキスト消費増）

### 案B：評価結果を別ファイルに分離

遺伝子ファイルは変更せず、評価結果だけをまとめた別ファイルを作る。

```
gene/001/evaluation.md  ← 全遺伝子のランキング一覧
```

- 各遺伝子ファイルへのWriteが不要になる（100回 → 1回）
- 遺伝子ファイル自体のReadも1回のBashで全文取得可能

### 案C：Bashでファイル結合してからまとめて評価

```bash
cat gene/001/*.md
```

でまず全ファイルを結合し、一度にClaudeに渡して評価する。
評価結果はCSVやJSONで一括出力 → evaluation.md に書き込む1回のWrite。

---

## 5. 結論

**主因**: gene-evaluatorが100個の個別ファイルを「一つ一つ」Read/Writeする設計

具体的には：
1. 100ファイル個別Read → 100ファイル個別Write の計200回以上のツールコール
2. 被りなしランキング制約で後半になるほど処理が複雑化
3. フォルダ構造（1遺伝子=1ファイル）がこの問題を構造的に引き起こしている

**推奨方針**: 評価結果を個別ファイルに書かず、`evaluation.md` 等の集約ファイルに一括管理する（案B + 案Cの組み合わせ）
